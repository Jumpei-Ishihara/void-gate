# SPEC-07 実装計画書 — スクロール3D「遊び方ガイド」再構成

対象: [SPEC-07](07-scroll-guide.md)（承認済み・推奨案採用: 総長約800vh / NAVIGATION存置 / フォームはCOMM末尾）
方式: これまでと同一の**テスト駆動**（テスト先行→実装→グリーン→目視→回帰→コミット→Pages反映→実機確認）

## 1. フェーズ構成と順序

```
Phase A (07a) ─→ Phase B (07b) ─→ Phase C (07c) ─→ Phase D (07d)
 基盤/回帰網          中核4章            入口/出口/結合         SP/性能/仕上げ
```

| Phase | SPEC | 内容 | 規模(目安) | ユーザー確認ポイント |
|---|---|---|---|---|
| A | 07a | Assets抽出・Tl/CHAPTERS・カメラパス・ページブルーム | 中 (+250行) | 見た目ほぼ不変。**ゲームの挙動が変わっていないこと**をPCで1プレイ確認 |
| B | 07b | 章02〜05（3D展開+コピー全文） | 大 (+600行) | **体験の核**。章02/03完成時点で中間スクリーンショット提示→トーン確認後に04/05へ |
| C | 07c | GATE/BRIEFING/NAVIGATION/LAUNCH DECK・HUD/コピー統一 | 中 (+350行) | 通しスクロール動画/スクショで全体レビュー |
| D | 07d | SP・reduced-motion・性能・run-all・ドキュメント同期 | 中 | **実機チェックリスト**（07d §5）の実施 |

- 各Phaseは独立コミット系列（`[SPEC-07a]`等）とし、Phase単位でPagesへデプロイ
  → ユーザーは実機/実ブラウザで段階レビューできる
- 問題発生時のロールバックはPhaseコミットのrevertで行う（ゲーム本体はMIG-03により全期間無傷）

## 2. 各Phaseの作業手順（共通テンプレート）

1. **Red**: `tests/phaseX.js` を先に作成（ACを機械検証に落とす。`VG.Tl._setT` による決定的駆動が中心）
2. **Green**: 実装 → テスト全通過まで反復
3. **目視**: 章ごとの代表tlでスクリーンショット確認（非表示タブ対策の直接レンダリング手順は既存踏襲）
4. **回帰**: 既存全スイート（phase1〜4 + 完了済みphaseA〜）を連続実行
5. **文書**: 該当SPECを Implemented→Verified に更新、逸脱があれば追記
6. **コミット/デプロイ**: push → Pagesハッシュ一致確認 → ユーザー確認依頼

## 3. テスト戦略

- **決定性が最大の武器**: SCR-05（純関数）により「_setT(t)→状態検証」で演出をユニットテスト化できる。
  章ごとに代表tl 4〜6点の期待値表（07bの表がそのままテスト仕様）
- 音のエッジ検出（B-06）は「往復で1回だけ」をSoundEngine._testのlastSfx計数で検証
- 性能はrender直計測（既存手法）+ スクロールRAF実測の2系統
- Phase Dで `tests/run-all.js` を導入し、以後は1コマンド回帰

## 4. リスク管理

| リスク | 対策 |
|---|---|
| Assets抽出でゲームの見た目が微変化（乱数消費順） | 抽出は関数移動のみで呼出順を変えない。万一変化しても「構造テスト」は通る設計。目視で岩の多様性を確認 |
| 章コンテンツ実装中にページが長期間「工事中」になる | Phase Bまでは旧セクションを残したまま**新章を末尾に追加**して開発し、Phase Cの結合時に一括差替え（本番が壊れた状態でデプロイされない） |
| スクロール量とテキスト読了速度の不一致（速すぎ/遅すぎ） | 章長(vh)を定数テーブル化し、実機レビュー後に数値だけで再調整できる構造にする |
| コピーのトーンが好みと合わない | 07bにコピー全文を明記済み — 実装前のこの段階で修正指示可能。実装後もテキストは定数テーブルで一括差替え可 |
| ページ側ブルームの負荷 | PCのみ+自動OFF(07a)。Phase A時点で実測し、予算超過なら以後のPhaseはブルーム無し前提で進める |
